# Design — `proxies` toolkit element + single-port-forward dev path

**Status:** approved, ready for implementation plan
**Author:** brainstormed in session, 2026-06-23
**Motivation:** post-demo cleanup item 2 — replace the eight individual `kubectl port-forward`s the demo UI backend relies on with a single proxy in front of every in-cluster backend the laptop UI touches. Goal: fewer moving parts on the laptop side, and a more resilient path that doesn't break when the laptop sleeps or one pod recycles.

This is a *toolkit* change (a new first-class deployable element) plus a small demo-side wiring change. It is not a runtime feature — the user-visible demo flow is unchanged. The dev/operator surface gets simpler.

---

## §1 — Topology

```
┌────────────────────────────── GKE cluster ──────────────────────────────┐
│                                                                          │
│   ┌───────────────────────────────────────────────────────────────────┐  │
│   │   HAProxy pod  (new element: `proxies.payments-proxy`)            │  │
│   │   ┌──────────────────────────────────────────────────────────┐    │  │
│   │   │ haproxy.cfg (from ConfigMap, generated by assembler)     │    │  │
│   │   │   :5432  → postgres-mainframe-proxy.mainframe-proxy:5432 │    │  │
│   │   │   :3306  → mariadb-analytics.mariadb-analytics:3306      │    │  │
│   │   │   :9094  → mainframe-to-gg-kafka.cdc-pipeline:9094       │    │  │
│   │   │   :8083  → mainframe-to-gg-connect.cdc-pipeline:8083     │    │  │
│   │   │   :10800 → mainframe-payments-gg8.<ns>:10800   (headless)│    │  │
│   │   │   :8080  → mainframe-payments-gg8.<ns>:8080    (headless)│    │  │
│   │   │   :9090  → prometheus.gg-prometheus-grafana:9090         │    │  │
│   │   └──────────────────────────────────────────────────────────┘    │  │
│   └──────────────────▲────────────────────────────────────────────────┘  │
│                      │                                                    │
│      ClusterIP Service `payments-proxy` (multi-port: 5432/3306/...)      │
└──────────────────────▲───────────────────────────────────────────────────┘
                       │
                       │  ONE  kubectl port-forward svc/payments-proxy
                       │       5432 3306 9094 8083 10800 8080 9090
                       │
                ┌──────┴──────┐
                │  Laptop UI  │   demo-ui backend reads localhost:<port>
                │  backend    │   (env vars unchanged from today)
                └─────────────┘
```

**Net effect.** `scripts/dev-port-forwards.sh` collapses from 8 forwards → 1. The UI backend's Kotlin code is unchanged — same `localhost:5432`, `localhost:9094`, etc. defaults; they just now exit the laptop via one tunnel instead of eight. GG `:10801` forward is retired because GG is load-balanced via the headless service through a single proxy listener.

**Why ClusterIP, not LoadBalancer.** Considered exposing the proxy on a GKE LoadBalancer for a truly zero-forward demo. Rejected: the LB IP would be public-internet-reachable, and today's demo credentials (`scripts/create-demo-secrets.sh` writes `payments-pw-replace-me`) can't be on the open net. ClusterIP keeps the proxy in-cluster, one `kubectl port-forward` brings it to the laptop, and the security posture is unchanged from today.

**Why one listener per GG node, not N.** The dev-port-forwards script today forwards both `gg8-0:10800` and `gg8-1:10800` so the thin client can use partition awareness (connect to the node that owns the key). A single proxy listener routes to the GG headless service, so each connection lands on a random GG pod — losing partition awareness. Acceptable for this demo: low concurrency, the generator caps at ~13.4k ops/sec, and one extra in-cluster hop adds <1ms. The schema *supports* per-pod listeners via an optional `pod_ordinals: [...]` filter for future demos that need partition awareness; the mainframe-payments demo doesn't use it.

## §2 — Element schema (`proxies.<name>`)

```yaml
proxies:
  payments-proxy:
    disabled: false
    infrastructure: mainframe-payments-gke
    k8s_namespace: payments-proxy          # own ns, like databases
    k8s_node_pool_template: default-gke-pool
    image:
      pull_from: <registry name>           # mirrors databases.<x>.image
      repository: haproxy
      tag: 2.9-alpine
      pull_policy: IfNotPresent
    pod_resources:
      cpu_request: 100m
      memory_request: 64Mi
      cpu_limit: 500m
      memory_limit: 128Mi
    listeners:
      - port: 5432
        target: { kind: database,       name: postgres-mainframe-proxy,  service: primary,        service_port: 5432  }
      - port: 3306
        target: { kind: database,       name: mariadb-analytics,         service: primary,        service_port: 3306  }
      - port: 9094
        target: { kind: cdc_connector,  name: mainframe-to-gg,           service: kafka,          service_port: 9094  }
      - port: 8083
        target: { kind: cdc_connector,  name: mainframe-to-gg,           service: kafka_connect,  service_port: 8083  }
      - port: 10800
        target: { kind: cluster,        name: mainframe-payments-gg8,    service: thin_client,    service_port: 10800 }
      - port: 8080
        target: { kind: cluster,        name: mainframe-payments-gg8,    service: rest,           service_port: 8080  }
      - port: 9090
        target: { kind: monitor,        name: pg-gke,                    service: prometheus,     service_port: 9090  }
    timeouts:
      deployment: 300
```

### Key shape decisions

- **`target.service` is a short name** (e.g. `kafka`, `kafka_connect`, `thin_client`, `rest`, `primary`, `prometheus`), not a raw k8s Service FQDN. Each element kind advertises a small fixed set of short names; the assembler resolves them to `<svc>.<ns>.svc.cluster.local:<port>` at deploy time. This keeps listeners portable across rename or namespace-move of backends, and surfaces typos at validation rather than at HAProxy connect time.
- **Listener external port = listener internal port.** No port translation. Nothing in scope needs it, and the constraint keeps `dev-port-forwards.sh` and the UI backend defaults aligned (the UI's `localhost:5432` is the proxy's `:5432` is the backend's `:5432`).
- **No `external_addresses` / `exposure` field yet.** Always ClusterIP. Adding `exposure: load_balancer` later is an additive schema change; the demo doesn't need it now.
- **GG handled by a single listener per port.** The schema *allows* multi-listener-per-cluster (via `target.pod_ordinals: [0, 1, ...]`) for future demos; the demo-config here doesn't use it. Demo accepts the partition-awareness trade for simplicity.

### Validation

- Referenced `target.kind` + `target.name` must resolve to a configured element of the matching kind.
- `target.service` must be one of the short names the kind advertises.
- No two listeners may share the same `port:` value.
- Listener `port` must be in `[1, 65535]`.

### Service-name registry (assembler-side)

| Element kind | Advertised short names | Resolves to (Service suffix) |
|---|---|---|
| `database` | `primary` | `<element-name>` |
| `cluster` (GG8) | `thin_client`, `rest` | `<element-name>` (headless) for both |
| `cdc_connector` | `kafka`, `kafka_connect` | `<element-name>-kafka`, `<element-name>-connect` |
| `monitor` (prometheus-grafana) | `prometheus`, `grafana` | `prometheus`, `grafana` |
| `monitor` (control-center) | `frontend`, `backend` | `frontend`, `backend` |

Out-of-scope element kinds (`infrastructure`, `cluster_monitoring`, `data_model`, `data_generator`, `assembly`) advertise no services — listing them as a `target.kind` is a validation error.

## §3 — Deployment lifecycle & Gradle surface

### Plugin code added

Mirrors the `databases` and `data_generators` patterns end-to-end.

| File | Role |
|---|---|
| `src/main/resources/schema/proxy.schema.json` | JSONSchema for one `proxies.<name>` entry — `listeners[]`, `target`, `pod_resources`, `timeouts`. |
| `src/main/kotlin/com/gridgain/demo/core/configuration/ConfiguredProxy.kt` | Strict-Kotlin DTO. No nullable types, no parse-time defaults; defaults live in the JSONSchema. |
| `src/main/kotlin/com/gridgain/demo/core/configuration/MigrateV15toV16.kt` | Additive migration — seeds `proxies: {}` when absent. |
| `src/main/kotlin/com/gridgain/demo/core/validation/ProxyValidator.kt` | Cross-element ref checks (see §2 Validation). |
| `src/main/kotlin/com/gridgain/demo/core/assembly/ProxySpecAssembler.kt` | Resolves each listener's `target` to `<svc>.<ns>.svc.cluster.local:<port>`. Consults the service-name registry. |
| `src/main/kotlin/com/gridgain/demo/core/spec/GkeProxySpec.kt` | Resolved, deployable spec. Carries `ResolvedImage`, `WorkloadScheduling.forElement`, `List<ResolvedListener>`. |
| `src/main/kotlin/com/gridgain/demo/core/deployment/ProxyDeployer.kt` | Apply Namespace, ConfigMap (generated `haproxy.cfg`), Deployment, ClusterIP Service. Wait Deployment ready. |
| `src/main/kotlin/com/gridgain/demo/plugin/tasks/DeployProxyAction.kt` | Walker action invoked by the Gradle task and by the assembly walker. |
| `src/main/kotlin/com/gridgain/demo/plugin/tasks/TeardownProxyAction.kt` | Walker action; **namespace-safe teardown** — does not delete a namespace shared with another element (the data-generator #3 fix). |
| `src/main/kotlin/com/gridgain/demo/plugin/DeployProxyTask.kt` / `TeardownProxyTask.kt` | Gradle task entry points. |
| `src/main/resources/templates/k8s/proxy/{namespace,configmap,deployment,service}.yaml` | Mustache templates. |

### Assembly walker integration

`AssemblyValidator` and `AssemblyElementDispatcher` get a new `proxy` element kind. The walker dispatches to `DeployProxyAction` / `TeardownProxyAction` exactly like the existing nine element kinds.

### Gradle tasks

- `deployProxy -PproxyName=<name>` (omit `-PproxyName` → all entries under `proxies`). Depends on `validateDemoConfiguration`.
- `teardownProxy -PproxyName=<name>`.
- Both honor `-PdryRun=true` (renders manifests under `<demoOutputDirectory>/k8s/proxies/<name>/` without applying), matching `deployDatabase` ergonomics.

### Assembly order

```yaml
assemblies:
  mainframe-payments:
    elements:
      - infrastructure          # mainframe-payments-gke
      - monitor                 # pg-gke
      - cluster                 # mainframe-payments-gg8
      - cluster_monitoring
      - data_model
      - database                # postgres-mainframe-proxy
      - database                # mariadb-analytics
      - cdc_connector           # mainframe-to-gg
      - data_generator          # payments-load
      - proxy                   # payments-proxy  ← NEW, last
```

The proxy is last in the walk because its validator requires the referenced elements to be present in config (it does *not* require them deployed — DNS resolves at HAProxy startup, and HAProxy retries until backends come up). Order-in-config doesn't affect runtime correctness; it keeps the assembly readable.

## §4 — Demo-side wiring

### `mainframe-payments-demo/src/main/resources/demo-config.yaml`

- Schema bump `15 → 16` (additive; `MigrateV15toV16` seeds `proxies: {}`).
- Add the `proxies.payments-proxy` entry verbatim from §2.
- Append `{ kind: proxy, name: payments-proxy }` to `assemblies.mainframe-payments.elements`.

### `mainframe-payments-demo/scripts/dev-port-forwards.sh`

The `FORWARDS=( … )` array collapses from 8 entries to 1:

```bash
FORWARDS=(
  "payments-proxy|svc/payments-proxy|5432:5432,3306:3306,9094:9094,8083:8083,10800:10800,8080:8080,9090:9090|All demo backends (HAProxy multi-port)"
)
```

`kubectl port-forward svc/payments-proxy 5432 3306 9094 8083 10800 8080 9090` — one process, all seven local listeners. The script's existing `check`/`watch` self-heal still works against the single forward. Resilience improves: the proxy's ClusterIP Service is more stable than the per-pod targets the old forwards used.

### Code clean-ups in `demo-ui/src/main/kotlin/com/gridgain/demo/payments/ui/`

- `config/UiConfig.kt`: drop the per-service "dev: port-forward this :Nxxxx" comments; one replacement note that all `localhost:*` defaults assume the `payments-proxy` is port-forwarded. The env vars themselves stay — operators can still override individual ones for split deploys.
- `services/{GridGainService,MainframeProxyService,MariaDbService,ConnectorControlService}.kt`, `tailer/KafkaTailerTap.kt`, `metrics/{GeneratorMetricsService,PrometheusCpuService}.kt`: error messages that say "start a port-forward" → "start the payments-proxy port-forward (`scripts/dev-port-forwards.sh`)". No behavior change.
- `demo-ui/frontend/src/components/GridGainPanel.tsx` (line 110): the operator-help blurb that mentions `kubectl port-forward` — updated copy.

### Memory updates

- `phase-2-unpause-easy-to-miss.md` — unchanged (still applies).
- New memory: "Single dev port-forward via `payments-proxy`" — short note that the multi-port forward is the new norm, and the rationale.

### What does *not* change

- The UI backend's env vars or their defaults.
- The CDC connectors' inbound configurations.
- The toolkit-handoff doc, except for one new row in the element-types table.
- Schema versions of `deployment.yaml` (no migration there — back-compat handled by per-spec defaults, as with `CdcConnectorStateBackCompatTest` / `DataGeneratorStateBackCompatTest`).

### Demo-side smoke

Captured as a manual checklist in `docs/track-coordination.md` under a new READY entry. No new automated demo tests; the existing `validateDemoConfiguration` regression covers schema/migration.

### End-to-end verification sequence

1. `./gradlew validateDemoConfiguration` — migrates v15 → v16 in place, stays green.
2. `./gradlew deployAssembly -PassemblyName=mainframe-payments` — proxy comes up last; HAProxy pod reaches `Ready`; multi-port Service has all 7 ports.
3. `kubectl get svc payments-proxy -n payments-proxy -o yaml` — confirms 7 ports; `kubectl exec` into the proxy pod, `nc -zv` each target FQDN.
4. Kill the old `dev-port-forwards.sh`; start the new (single-line) version. `lsof -nP -iTCP:5432`, `:3306`, `:9094`, `:8083`, `:10800`, `:8080`, `:9090` all LISTEN on the laptop.
5. `./gradlew :demo-ui:run` — UI loads; phase 0 reads from Postgres; phase 2 dump+load+unpause cycles through the proxy; tailers receive Kafka events through the proxy.
6. `kubectl delete pod -l app=payments-proxy -n payments-proxy` — proxy pod restarts; **kubectl port-forward survives the restart** (the Service is stable), UI reconnects per existing retry logic. This is the resilience win.
7. `./gradlew teardownAssembly -PassemblyName=mainframe-payments` — proxy torn down with everything else; namespace removed cleanly; no orphaned LB / IP (none was allocated).

### Risk register

- **GG thin client without partition awareness.** A single listener (load-balanced via kube-proxy) means each connection lands on a random GG pod. For this demo's traffic shape (low concurrency, 5 customers, generator caps ~13.4k ops/sec) the impact is one extra in-cluster hop, <1ms. Documented in the new element's `docs/proxies.md` and in the demo's risk callouts.
- **HAProxy ConfigMap generated from `listeners[]`.** Any drift between the generated `haproxy.cfg` and what HAProxy expects (config-format breaking changes across versions) blocks deploy. Pin the image tag (`haproxy:2.9-alpine`) and lock the format with the `ProxyTemplateModelTest` fixture.
- **DNS resolves at HAProxy startup.** If the proxy somehow comes up before a backend Service exists, HAProxy logs `Could not resolve…` and retries. The assembly's element ordering puts the proxy last so this is rare; HAProxy's retry policy self-heals if it does happen.
- **`kubectl port-forward` of 7 ports in one process.** kubectl supports it natively; if the process dies, all 7 listeners die together. That is an *improvement* over today (8 independent processes that can silently fail one at a time, leading to "tailers empty / Connect not reachable / GG hangs" debug rabbit-holes). The existing self-heal in `dev-port-forwards.sh` continues to work against the single process.

## Out of scope (acknowledged, deferred)

- **LoadBalancer exposure of the proxy** (public IP, zero forwards). Easy to add later via an additive schema field (`exposure: load_balancer` with optional `loadBalancerSourceRanges`); requires a credential-rotation conversation that's not on the table for v1.
- **TLS termination.** All proxy listeners are plain TCP. Fine for in-cluster + single-laptop-port-forward; would matter if we ever exposed externally.
- **HAProxy stats / health endpoints.** HAProxy supports a stats listener; not configured. Easy add if useful for operator UX.
- **Multi-proxy demos.** The element type is per-name, so a future demo with two proxies (e.g. one per region) works without further schema work; not exercised by this demo.
- **Auto-discovery of backends** (proxy listeners derived from the assembly's other elements). Considered; rejected for now. Explicit listeners keep the element standalone, match the toolkit's "elements don't peek at other elements at deploy time" convention, and surface intent (which ports are externally relevant) clearly.

## Open questions

None at design time. All v1 decisions have been made:

- **Element type** — new first-class `proxies` (not demo-local YAML, not an overload of an existing kind).
- **Proxy software** — HAProxy (TCP mode, 2.9-alpine, ConfigMap-supplied config).
- **Listener shape** — element-ref via kind + name + service short-name + service_port.
- **GG fan-out** — schema supports both; demo picks single-listener-per-port.
- **Exposure** — ClusterIP, one port-forward to the proxy.
- **Scope** — all 7 distinct backends (Postgres, MariaDB, Kafka EXTERNAL, Kafka Connect, GG thin-client, GG REST, Prometheus). The 8th forward today (GG node-1) is retired via the headless-service listener.

Track-coordination implications: this is a *toolkit* deliverable (new element type) plus a demo-side wiring change. A READY/INTEGRATED pair in `docs/track-coordination.md` will mark the handoff, mirroring how the data-generator and cdc-connector elements landed.
